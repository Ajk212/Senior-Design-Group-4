# Senior-Design-Group-4
This repository holds the code for any software used in Group 4's Senior Design project

HandSense is a embedded system specializing in recognizing hand gestures and classifying them so computer systems may be controlled via simple and intuitive motions.

1. Introduction
Many computers commonly use peripherals such as mice, keyboards and touchscreens to allow users to interact with their device and send inputs. While these devices are popular by allowing effective and precise inputs, they are also not equipped to be accessible by all types of users. For those with disabilities, devices like these can act as barriers, limiting the amount of accessibility the user has on the device. By introducing the ability of providing computer input via gesture-based control, we can ensure that the range of users capable of interacting effectively with the system is broadened to accommodate a variety of disabilities and circumstances.
The HandSense gesture recognition glove is designed to address these challenges by serving as a compact, multi-sensor wearable interface capable of fully navigating a personal computer system (PC) with only the use of a single hand. Unlike the other gesture recognition devices which are currently available on the market, which often narrow focus into translating American Sign Language (ASL) into spoken English, HandSense provides a broader use case. Users can navigate through a computer system, as they could with a traditional mouse, along with providing text input via ASL gesture recognition. This framework allows for diverse applications to be created to ensure optimal and effective user experiences.
When looking into the device at a technical level, HandSense integrates a suite of sensors to accurately detect user gestures. A microcontroller houses a lightweight machine learning model for real-time recognition of gestures, the signal of which is sent via Bluetooth for seamless connectivity with many modern computers. An attached indicator provides the user with feedback on the device’s connectivity, battery level, and the currently selected mode. The lightweight design of the glove allows for efficient use without compromising the performance of the onboard model, or dampening other factors such as battery life, signal speed and accuracy.
To meet these needs, we designed hardware consisting of flex sensors, dual IMUs, a Li-Po battery with protection circuitry, and an ESP32-C6 microcontroller, complemented by firmware capable of running a one-dimensional convolutional neural network and software for real-time gesture recognition. Experiments included SPICE simulations of sensor circuits, PCB connectivity checks, and functional testing of gesture recognition with haptic and visual feedback. Results showed that the glove successfully recognized 15 predefined gestures and enabled full PC navigation with accuracies of over 85% and responsiveness of an average of 111ms, while maintaining a battery life of over eight hours, meeting most design requirements for usability, portability, and runtime.
While the device is primarily suited for allowing accessible options for users with disabilities, its robust design and versatile software create the opportunity for broader applications. Ordinary users will be able to take advantage of the glove for tasks such as lecturing, conducting presentations, multi-tasking or general at home convenience. This combination of accessibility and adaptability provides HandSense with the ability to be an assistive technology along with a practical everyday tool.

2. Background
Gestural interfaces have become increasingly important in human-computer interaction (HCI) and Internet of Things (IoT) systems. Unlike voice or touch input, gestures allow for silent, intuitive, and effective control, which is valuable in a noisy environment or accessibility contexts such as ASL. Applications in assistive technology for the deaf, hard-of-hearing community and compatible with ASL has been a longstanding research area, however they often rely on vision-based methods using cameras and computer vision algorithms, which are limited by lighting conditions, backgrounds and privacy concerns. Wearable standalone sensor-based approaches provide a more robust alternative.
ASL is the primary language for over half a million people in the United States, with millions more using it worldwide (Mitchell, 2022). Despite its widespread use, communication barriers persist between ASL users and non-signers. Technological solutions for ASL translation aim to bridge this gap, but many systems remain impractical outside perfect laboratory settings. Vision-based recognition, while advancing rapidly with deep learning, raises concerns around privacy and requires significant computational power. Wearable glove-based approaches provide a more portable and privacy-preserving alternative. However, existing designs often suffer from bulkiness, limited battery life, or reliance on external processing devices.
SignAloud is one of the earliest notable smart gloves developed to translate ASL gestures into spoken English (Taggart, 2024). It integrates multiple sensors placed on the hands and wrists to measure hand positions and movements, transmitting data wirelessly via Bluetooth to a computer for processing. A statistical regression model based on a neural network interprets the sensor data and outputs corresponding English words. While SignAloud demonstrated the feasibility of wearable ASL translation, it heavily relies on external computing resources for inference, limiting portability and real-time responsiveness. Additionally, its dependence on bulky hardware and the need for continuous wireless communication constrain its use in everyday, standalone scenarios. 
Researchers at UCLA developed a wearable glove equipped with stretchable sensors extending along each finger, capable of detecting complex hand and finger motions (Chin, 2020). A compact circuit board on the wrist collects sensor signals and transmits them via Bluetooth to a smartphone, where machine learning algorithms translate the gestures into words, achieving approximately one word per second. This design improved wearability and demonstrated a low-cost approach to gesture recognition. However, the need for continuous smartphone connection for translation, limited recognition speed, and lack of tactile feedback reduce its suitability for real-time ASL communication or broader IoT applications. 
The University of London’s smart glove integrates five flex sensors, one per finger and an accelerometer positioned on the back of the hand (Hunt, 2015). The glove operates on a small coin-cell battery, prioritizing lightweight design and low power consumption. Initial data processing occurs on-device before being transmitted via Wi-Fi to a companion mobile application for gesture recognition and interpretation. Although compact, the glove’s limited power source restricts computational capability and runtime, forcing reliance on external devices for full translation. The use of Wi-Fi also introduces latency and potential connectivity issues, making this approach less robust in mobile or offline environments.
UCSD’s smart glove was designed as a low-cost (<$100) ASL-to-text system using two flexible strain sensors per finger and one on the thumb (Oberhaus, 2017). Sensor data is processed locally, and the recognized gestures are transmitted via Bluetooth for display as text. The design successfully demonstrates affordability and on-edge preprocessing but remains limited to alphabetic letter recognition rather than full ASL words or phrases. The system lacks multimodal feedback (such as haptics or visual indicators) and does not incorporate onboard learning models, constraining adaptability and user interactivity. (Oberhaus, 2017)
The SignLanguageGlove project utilizes five flex sensors, an accelerometer, a microcontroller, and a numeric display to interpret gestures (Hussain, 2015). Data is sent to a PC via serial communication for processing, then relayed to a smartphone app over Wi-Fi. While the glove offers basic gesture detection and visualization, its dependency on multiple external devices for computation and output undermines portability. The relatively high device cost (around $386 per unit) also limits scalability and accessibility. Furthermore, the absence of onboard intelligence or feedback mechanisms reduces its potential for autonomous and intuitive operation.
HandSense addresses these limitations by integrating five flex sensors and dual IMUs with an ESP32-C6 microcontroller capable of running 1D convolutional network model locally via TensorFlow Lite Micro. This eliminates dependence on cloud or external processing, enabling low-latency recognition. Unlike prior gloves, HandSense incorporates haptic motors for tactile confirmation and an indicator for real-time status, supporting both accessibility (ASL translation) and general-purpose IoT control.	
3. System Requirements
The primary use of the HandSense system will be to remotely operate a PC by simple hand gestures so that the user can be relieved from sitting in front of a PC operating a keyboard or mouse. Once the user wears the glove and turns on the device, the device will negotiate with the PC and establish a connection. System configuration to establish a connection requires only one initial setup. Once the user is able to connect to the PC, the device will record the settings so that the user can simply turn on the device and instantly use it going forward. The user will receive haptic feedback based on the operations performed on the PC. The user will also be able to type letters in text fields using hand signs following the American Manual Alphabet of the American Sign Language specification (referred to as ASL going forward). 
3.1 Physical Requirements
Physical requirements relate to the feel of the device when the user is using the glove and operating a PC with it. For the system to become a valid alternative to existing devices on the market, it must have a pleasant user experience where users are not forced to endure discomfort. Requirements that fall in this category are focused on the features that provide the user with comfortable user experience. 
3.1.1 	The product must fit to a human hand so as not to obstruct the movement of the hand and arm while in use
The product must allow the user to perform daily movements such as opening and closing one’s hand, grabbing, and picking up objects while the device is in use.
3.1.2	The product must be of a size and weight that is easy to carry around for an average adult on one hand
Given the average size of an adult female hand is 6.8 inches (17.3cm) long and 3.1 inches (7.9cm) wide, it must be smaller than this dimension. Given popular handheld devices such as an iPhone weighing around 170 grams, it must be of equivalent or less weight. The user must be able to wear the device continuously throughout the day without feeling any fatigue.
3.1.3	The product must fit onto the user’s hand with no energy required by the user to hold the product in place during continued use
The product should not feel bulky or uncomfortable for the user to continuously use it for over 8 hours. The temperature of the product should not get high enough to cause discomfort, thus temperature should not exceed 40 degrees Celsius (Craig, 2009).

4. Design Constraints: Standards and Impacts
A design constraint is a limitation imposed by an external source, independent of consumer requirements. Unlike design requirements, which are based on the needs of the user (e.g., “the glove must recognize ASL gestures”), design constraints are dictated by standards, regulations, or unavoidable external conditions (e.g., battery safety regulations, wireless communication standards). They determine the internal operating mechanism of the system that does not directly relate to the user experience. The HandSense system is intended to operate as a consumer electronics device. Thus, in addition to features the customer “require”, it must comply with the various regulations that ensure safety and environmental accountability. The following subsections outline the external constraints and the non-technical impacts relevant to the design of HandSense. 
4.1 Design Constraints
4.1.1 Time
Time is a key design constraint in all aspects of life, especially in this semester project, since students would have to juggle other coursework and other responsibilities that they might have. With only a few months to design, build, integrate and test the glove, the team must correctly assess the scope of the project and draft a schedule that considers the total amount of time in the term remaining to complete the project, listing out the progress that each team member should be making every week. The team should also make sure that the plan can be followed accurately and still have a buffer for unforeseen situations. The time constraint also forces the team to make sure that necessary components ordered on time accounted for lead time to not impede the overall progress of the project. 
4.1.2 Budget
The project budget of $200 imposes strict limits on the glove’s design. While consumer assistive technology devices may be priced in the hundreds or even blocked behind a quotation, the team must balance affordability with performance. This affects component choice, prioritizing lower-cost IMUs, flex sensors, and microcontrollers over specialized or custom-built hardware. For example, the original design included a flexible PCB for the finger and palm which has been reassessed by the team and decided to be too expensive. The budget also forces the team to be more careful with components used during prototyping, making sure that the components wouldn’t be broken during prototyping and requiring it to be reordered, increasing the budget.
4.1.3 Manpower
The team working on this project consists of three members, limiting the amount of design, coding, and testing that can be accomplished. With gesture recognition, Bluetooth communication, and device feedback all requiring development, the lack of manpower restricts the number of advanced features that can be implemented. In an industry setting, larger teams could add power optimization, and ergonomic refinements. 
4.1.4 Physical - Marketplace Product Requirements
For consumer electronics sold in the European Union, the glove must meet the Restriction of Hazardous Substances Requirements (RoHS). This limits the design of components and solders free from harmful levels of lead, cadmium and other toxic materials. Even though this project will not enter mass production, RoHS-compliant parts help future-proof the design. The device must also comply with Underwriter Laboratories (UL) standard 60065. As the device’s intended market is for ordinary consumers, distributors such as Amazon require electronic products sold on their market to be compliant with one or more UL standards.
4.1.5 Electrical - Battery and Charging Safety
As a wearable device, HandSense relies on a lithium-polymer (Li-Po) rechargeable battery. International standards such as IEC 62133-2 and UL 2054 regulate battery safety, requiring protection circuits against overcharging, short-circuiting, and thermal runaway. Another standard to uphold is the 10 Code of Federal Regulations (CFR) Part 430 since the device includes a battery recharging circuit and is intended to be sold to the public. This constraint affects the choice of battery cells, charging circuits, and enclosure design to ensure safe consumer use. 
4.1.6 Electrical - Wireless Communication Standards
HandSense uses the Bluetooth Low Energy (BLE) capabilities on the ESP32-C6 to transmit gesture data to PCs and other IoT devices. BLE devices must comply with Federal Communications Commission (FCC) Part 15 regulations in the United States, which limit transmission power and require devices to operate in the unlicensed 2.4 GHz ISM band without causing harmful interference. This constraint restricts wireless modules, and maximum communication range of the glove to the receiving device.
4.2 Impacts on Non-Technical Contexts
4.2.1 Battery Choice Impacts
To power the wearable glove, the team considered multiple alternatives, including a Li-Po battery and coin cell battery. The Li-Po batteries provided high energy density and high discharge current, which is required to power the glove’s active sensors and motors. This choice impacts welfare and safety, because Li-Po batteries can overheat, swell, or catch fire if mishandled, requiring proper charging circuitry, insulation, and protective enclosure. It also has impacts on the environment, since Li-Po recycling must be managed carefully. On the positive side, it allows ergonomic usability by keeping glove weight reasonable and maintaining sufficient runtime. The other option, coin-cell batteries have very low current capability (typically <5–10 mA continuous) and low capacity (few hundred mAh). This makes them unsuitable for powering high-current devices like motors or frequent wireless transmissions. Using coin cells would significantly limit the glove’s functionality, prevent haptic feedback and reduce BLE range or stability, negatively affecting user experience, accessibility, and device effectiveness. The benefits of coin cells are that they are safe, non-toxic, and easily replaceable, which slightly improves environmental impact. To summarize, Li-Po batteries are necessary to meet performance requirements but introduce safety and environmental considerations, whereas coin-cell batteries are safe but insufficient to meet power and performance requirements.
4.2.2 Diversity, Equity and Inclusion
To ensure a device that provides a positive impact in diversity, equity and inclusion, the HandSense design was built to allow accessibility for those unable to operate conventional computer equipment such as a keyboard or mouse. The one-handed design minimizes the physical effort required, making input both simple and intuitive. By incorporating gestures based on the ASL alphabet for text entry and straightforward hand movements for navigation, the device promotes equity by creating an alternate input method that recognizes and validates diverse forms of communication. The software package contributes to these contexts as well by providing a means to connect via Bluetooth effortlessly. Attention to comfort and usability during prolonged use ensures that users are not excluded due to fatigue or physical strain. By allowing those who cannot conventionally interact with computers with a comfortable means to do so, we hope to reduce the number of individuals incapable of working in the digital space.
5. Summary of Design
5.1 Design Alternatives Considered in Conceptual Design
There were two designs which were considered in the Team’s conceptual design: one design focusing on the accuracy of gesture tracking and minimizing latency to provide the best user experience and another focusing on reducing power consumption and improving energy efficiency. Both concepts shared the communication method of using Bluetooth Low-Energy to communicate between the device and PC. The design concepts differed in the sensors used to capture finger movements, choice of power source and related power management systems, and machine learning algorithm to perform inference on collected sensor data. 
For sensors, the team considered the use of resistive-flex sensors to track finger bend angles, resistive touch sensors to capture “pinching” movement when tapping the tip of the thumb with the other four fingers, IMUs to track the movement of the hand. In addition to the sensors themselves, the team considered auxiliary circuits that would aid in improving the measurement accuracy such as the Wheatstone bridge connected with an instrumentation amplifier, as well as an external Analog-to-Digital Converters (“ADC”) with simultaneous conversion capabilities to reduce latency of the microcontroller by freeing up clock cycles of using an integrated ADC on the ESP32-C6 microcontroller. 
For the power source, the team considered the trade-off between cost and complexity against user-experience and energy efficiency. A lithium-ion polymer battery combined with battery protection, battery charging, and USB power delivery controller circuit was compared to a coin battery with a bespoke linear low-dropout voltage regulator using discrete components and battery watch-dog firmware. Based on the power source and expected current draw of the circuit, three microcontrollers with integrated Bluetooth capabilities were compared for their energy efficiency and clock speed. 
Two alternatives were compared for analysis for the machine learning algorithm to recognize gestures on the edge: Long-Short Term Memory (LSTM) machine learning model and a 1D-Convolutional Neural Network (1D-CNN). The LSTM model, commonly used for inference using time-series data achieving high accuracy. However, it required large number of parameters to be stored in memory. Whereas the 1D-CNN model was lightweight and fast, requiring less memory to perform inference but with lower accuracy.

5.2.1.1 Sensor Reading Circuit
Flex sensors change resistances based on the bending of the sensor.  With some voltage applied, the change in bend angle can be picked up by the microcontroller by measuring the voltage across the sensors given the shift in resistance. By placing the flex sensors along each finger, the circuit allowed the device to capture the precise finger movements. 
To measure finger movements, a flex sensor (FS-L-055-253-MP with rated resistance of 25kΩ ± 30%) was attached to each finger. An additional touch sensor (SEN0294 with a rated resistance range of 200Ω - 100kΩ) was placed on the thumb to allow the user to control the device without the use of physical buttons. Each sensor connected to a Wheatstone bridge and an instrumentation amplifier (INA2331). The output of the instrumentation amplifier then was connected to the GPIO pins of the ESP32-C6 which had integrated ADC functions. This circuit was chosen to improve accuracy of sensor reading as the original scope of the project included the ability to capture a wide range of hand gestures, which include the capability to recognize the subtle differences in ASL. Though the circuit required three instrumentation amplifier ICs as the system had a total of six sensors connected to the ESP32-C6 and each IC only could accept two inputs, the overall size the sensor reading circuit was small enough to comfortably fit a human hand as shown in Figure X2. 
5.2.1.2 Battery and Battery Management Circuit
A rechargeable Li-Po battery with a capacity of 900mAh was selected as it allowed the design team to satisfy the operational requirement (section 3.1.4) while addressing the impacts to the environment (section 4.2.1), compared to using a non-rechargeable battery. With the estimated current draw in the range of 43mA to 218mA as shown in Table 1, the battery size was determined sufficient to run the device for over 8 hours. The battery protection and charging circuits were  designed so that they comply with relevant regulations required for hand-held electronic devices listed in sections 4.1.4 and 4.1.5. Due to the ESP32-C6 requiring 3.3V to operate and the voltage regulator requiring at least 3.8V to produce a steady 3.3V supply, the Li-Po battery required a higher charge voltage than 3.8V throughout the battery discharge process. Therefore, a 7.4V Li-Po battery with a charge voltage range of 6.4V to 8.4V was chosen as it could supply sufficient voltage throughout the charge cycle.

Since the device is worn directly on a user’s hand, it was critical that the battery charging and discharging cycle is managed to ensure safety of the user. The battery protection circuit used ICs specialized in Li-Po battery protection with three layers of safety mechanisms to ensure safe operation. 
The first protection circuit connected with the Li-Po battery in series was an in-line fuse (ERB-RE1R50V rated at 1.5A, 32VDC). It is a simple fuse that shuts off when a short circuit in the discharge circuit is detected, or in this case a current above 1.5A. This fuse acts as the last line of defense when a fault state is detected when both IC and firmware protection schemes fail to cut off power from the battery. Since the expected maximum current draw was around 218mA per table 1, a fuse rated at 1.5A was deemed sufficient to prevent faults based on Li-Po battery circuit designs that recommend setting a fuse that is rated to 150% of the operating current. 
The second protection circuit consisted of a dedicated battery protection IC (BQ29700DSER). The controller circuit of this IC can cut off the negative node of the battery by swishing a MOSFET driver when detecting either over voltage or under voltage. The over voltage lockout was set to 8V and the under-voltage cutoff is set to 6V given the Li-Po battery specification of discharge voltage of 6.4 – 8.4V.
The third protection circuit consisted of a gas gauge IC (STC3100) with an internal temperature sensor to monitor both available battery capacity and battery temperature. This IC was chosen as it had an internal ADC that converts the current, charge voltage, and temperature readings into digital signals communicated to the microcontroller by I2C, obviating the need for additional sensor amplification circuits. By programming the microcontroller to monitor the temperature and current, the microcontroller can cut off power from the firmware prior to damaging hardware.

5.2.1.3 USB Power Delivery Controller and Battery Charging Circuit
As the USB Type-C connector has become the industry standard for consumer electronics due to its versatility in both data transmission and power delivery. Therefore, the device used this connector for both charging and programming the microcontroller. There are various USB power delivery standards that rely on the USB type-C which must be negotiated between the receiver and power connector (Triggs, 2025). For this purpose, CYPD3177 was chosen based on its ability to manage both output voltage and current using external voltage dividers in addition to its low operating current of 10mA suitable for battery powered devices. The resistance values were set to support USB power of up to 20V and a current draw capped at 100mA (CYPRESS).
Similar to the battery protection circuit, careful attention was given to the battery charging circuit to ensure consumer safety, chiefly to prevent overheating while charging and to safely switch between battery power and external power. Li-Po Battery Charge Management Controller IC (MCP73781) was selected as it is commonly used in portable electronics that use USB port power. This IC was selected based on the feature that allowed switching between battery power and USB power. This obviated the need for an external MOSFET circuit to switch between the two sources. 
5.2.1.4 Haptic Feedback
A 3V DC motor (VW0620AB001U) with a max voltage rating of 3.3V was chosen based on the power supply and requirements of detailed haptic feedback from the motor. Combined with a motor driver IC (DRV2605), the motor controller circuit was placed on a PCB located on top of the index finger with limited area. The integrated IC operating on a PWM signal saved space compared to designing a motor controller circuit made with discrete components. 
5.2.1.5 Microcontroller
To satisfy both the low latency (section 3.1.7) and portability requirement (section 3.1.2), the microcontroller had to have sufficient computing power to translate sensor readings into PC commands with integrated wireless communication capability. For this purpose, ESP32-C6 was chosen as it has both the memory space to perform inference on machine learning algorithms and the clock speed to provide low latency responses when consecutively performing calculations and Bluetooth communication. The ESP32-C6’s 22 GPIO pins allowed connecting all peripherals used in this system onto a single chip. In addition, the 512kB of RAM, and 320kB of ROM combined with a clock speed of 160MHz was particularly suited for performing inference on larger machine learning models such as LSTM and 1D-CNNs.

5.2.2 Firmware
The Firmware system consists of the following subsystems: OLED display for user-experience, sensor reading, battery tracking, haptic motor control and Bluetooth connectivity on the glove and the client side.
5.2.2.1 OLED Display Subsystem
A 0.96” OLED display would be used to communicate the Bluetooth connection status, remaining battery life, gesture recognition, status of the glove and mode of the system (such as locked or navigation / text input). A 128 by 64 pixel 0.96” OLED display was chosen as it has sufficient space to place all the required information while operating by a 3.3 – 5V DC voltage at 5mA current draw. The SSD1306 is a common OLED display IC used with microcontrollers. The custom driver was written to update and clear the display. The display is broken up into 8 pages(rows) and 128 columns(pixels). The OLED driver also stores the bitmap for basic characters, Bluetooth and battery life symbols are saved to be called and drawn onto the display. This implementation removes the need for pixel-by-pixel manipulation and is more efficient.
5.2.2.2 Sensor Reading Subsystem
The firmware for HandSense continuously acquires motion and flexion data from five flex sensors, two IMUs, and one touch sensor to capture hand gestures with high accuracy and low latency. The communication between all the different sensors and the microcontroller is managed through I2C 1 master multiple slave mode. The flex sensors and touch sensor are read by the ESP32-C6’s ADC channels at a sampling rate of approximately 32 kHz, providing sufficient temporal resolution for finger movement tracking while maintaining low power consumption. The flex and touch sensors’ voltage readings are first passed through a Wheatstone Bridge and Instrument Amplifier to get stable gain. The ADC onboard the ESP32-C6 then calibrates the values by taking samples of the raw readings and applying a curve fitting scheme to get a better voltage value. 
Each IMU communicates over the I²C bus and operates at 32 kHz, capturing both acceleration and angular velocity data for wrist and finger orientation. The IMU has been evaluated using different acceleration and gyroscope sensitivities and a configuration of Data from all sensors are synchronized using timestamping and stored in a sliding buffer for preprocessing before being passed into the onboard gesture recognition model. Preprocessing includes noise filtering, normalization, and feature extraction to ensure robust recognition under varying conditions. The touch sensor operates as a digital input with interrupt capability, providing immediate response for gesture confirmation or mode switching. By performing all preprocessing and inference locally, the system minimizes Bluetooth communication overhead and ensures real-time responsiveness. 5.2.2.3 Battery Tracking
Using the STC3100 battery monitoring IC, the state of charge of the battery could be tracked and monitored. The hardware coulomb counter and current draw reading was used to implement an approximation of the battery that would be updated and saved into NVS(non-volatile storage) on the ESP32-C6. This would allow battery runtime to persist between power cycles and during power loss. The calculated charge state would then be updated on to the OLED for the user to know when to charge the device.
5.2.2.4 Bluetooth Connectivity Subsystem
Given the option of using either the integrated Wi-Fi or Bluetooth peripheral of the ESP32-C6, Bluetooth was chosen based on the amount of data transferred and proven capability on wearable devices available in the market ( (Chin, 2020) , (Hunt, 2015)). Another metric that was used in consideration of choosing Bluetooth was that BLE (Bluetooth ) is the more energy-efficient protocol. 
A custom GATT service was built for bidirectional Bluetooth communication. The Bluetooth subsystem is used to transmit the gesture inferred from the model from the device to the computer. The length of the instruction set would depend on the number of actions available to the user, however, it is not expected to exceed the 800Kbps rated bandwidth of the Bluetooth protocol. The software will be configured to run in the background of the PC, similar to a Bluetooth keyboard or mouse. Once the script on the host PC is running and detects that the glove is turned on, it will automatically initiate paring and authentication of the Bluetooth-enabled ESP32-C6 on the glove as shown in Figure 9. The main loop of the program also uses the Bluetooth submodule to send a gesture string that was inferred from the model to the PC client to be executed. The custom service allows it to be sent as a notification so that the client does not need to constantly read from the RX characteristic. After the execution of the command corresponding to the gesture sent, the client would send back an “ACK” string, telling the glove to start gesture recognition loop again. During model training, a data pipeline was set up to stream the sensor data to client for recording gesture datasets to train the model. The data was sent as a frame including all the sensor readings in that instance with the timestamp and metadata. This allowed for more efficient training times. 5.2.3.1 Machine Learning Model
For gesture recognition and command execution, a 1-D neural network was implemented onto the ESP32-C6 microcontroller. This neural network is lightweight and fast, although it does not carry the same levels of accuracy and memory as the LSTM model used in initial stages of the project’s development. This neural network excels in Human Activity Recognition (HAR), meaning that the model is capable of observing real-time sensor data to make informed predictions regarding gestures performed.  The major drawback of this neural network is its limited precision and memory. These limitations posed a potential issue, threatening to reduce the number of available gestures to allow for greater accuracy. Evaluating the model revealed that the defined set of 15 gestures retained high accuracy (around 95%). This model was concluded to be the best choice for our prototype, allowing us to maximize performance while maintaining a small file size so the model could be formatted into a TensorFlow-Micro Lite file. This format conversion allowed the model to be placed onto the glove’s onboard ESP32-C6 processor and run all predictions locally, reducing latency and improving user experience. Onboard the glove, the model is presented with data from each of the five flex sensors and two IMUs. After a sliding window is filled with data, it tries to generate a prediction by comparing the received data against the data used to conduct its training. If the model predicts a gesture with a certainty above 85%, it classifies it as performed, sending the gestures ID over Bluetooth to the PC client. After this data is sent, it waits for a return signal, marking successful communication between devices.

5.2.3.2 PC Client
The PC client used in the final design consisted of two key parts, a frontend and backend system. The frontend system presents the user with a GUI, an interface allowing for user customization of gesture commands and control over the establishment of a connection with the device. By choosing items from drop-down boxes, the user is able to rebind performed gestures with a pre-selected list of commands, such as click, minimize window, increase volume and more. This client is lightweight and utilizes minimal resources, allowing the program to run in the background without issue. After the device is connected with the client, the GUI updates relevant connection information and through an advanced view, allows users to view real-time communication results being sent. This advanced view through the terminal allows for testing to be conducted while the device performs normal operations. The GUI gesture configuration interface is shown in figure 12. 


Figure 12 – PC Client GUI

The second part of the PC client is the backend system. This part of the system facilitates all communication and command execution for HandSense. Using a Bluetooth Low-Energy protocol, the PC client is capable of sending and receiving data from the device with minimal latency. After receiving a predicted gesture from the device, the backend communicates with the frontend system, checking the received gesture against a table of bound commands. After acquiring the associated command, logic is conducted to properly execute the command and reply to the glove with an acknowledgment statement, signifying that the command was successfully completed and that it is ready to receive the next gesture identified. This backend system remains lightweight along with the frontend, ensuring minimal delay in gesture execution and minimal system impact, allowing for long a duration of use without any issue.
5.3 Sustainability Considerations  
5.3.1 Hardware Sustainability Considerations
The rechargeable Li-Po battery used in this design would reduce non-rechargeable battery waste by allowing users to charge the battery throughout its life span. In addition, the primary PCB enclosure would be designed to allow the user to swap the batteries if the batteries have degraded after going through repeated charging cycles. 
Sensitive hardware such as sensors, PCB, and batteries were enclosed in a rubber coating and then woven within the fabric of the glove, preventing damage from external environments. This would contribute to extending the product life span.
5.3.2 Firmware Sustainability Considerations
Firmware sustainability was addressed by only sending out Bluetooth commands to the PC when the model has 80% confidence in gesture detection, this is to prevent energy wasted sending the command via Bluetooth and the user having to roll back the keyboard input or undo the navigation command when the gesture detected isn’t confirmed. Another way to minimize energy consumption was to put the microcontroller in low-power mode and periodically checking the sensors to wake the system up.
5.3.3 Software Sustainability Considerations
When developing the conceptual framework for the software, sustainability was a large consideration to ensure that the device minimally affects the environment and society in a negative way. The choice of using an LSTM machine learning model or a 1-D convolution model was made due to the models having low-power capabilities. Both models are lightweight enough to run off a microcontroller and remain power efficient even when under load. The LSTM model specifically is able to be condensed into a package which eliminates all excess power draw, making it a perfect choice to use for this device. By keeping power low, the device lasted longer and reduced emissions it generated.

The results showed a semi-linear function that could approximate the behavior of sensor readings, thus, finger movements. The output voltage from the touch sensor was similarly measured. However, the touch sensor did not require accurate reading of the data as it would act primarily as a button. Therefore, the team only confirmed that a pinching movement of the finger generated enough force required to activate a response.

6.4 Software Systems
To ensure total software system functionality, testing was conducted at each major design stage. The main evaluation criteria for this system will be the model’s gesture recognition accuracy. Maintaining a high accuracy ensures improved user experience along with higher efficiency within the codebase.
6.2.1 Preliminary Model Testing
While the development of the hardware was underway, preliminary testing focused on training and evaluating the LSTM machine learning model to ensure its suitability for real-time data inferences as well as integration with the final system. Due to physical sensors being unavailable at this time, the model was initially trained through simulated sensor data. This simulated data provides a controlled environment that allows for rapid testing, refinement, bug detection, and aided in overall refactoring. Although the simulated data did not fully resemble the complex data sets acquired from real sensors, they served to validate and test the gesture recognition framework. By conducting this preliminary set of tests, development focused on refining the accuracy and certainty of gesture detection, resulting in a model that is more precise and ready to be implemented with the hardware systems. This stage of testing involved simple gesture sets being generated to train the model, followed by providing the trained model with a varied set of data to validate confidence strength and accuracy.
6.2.3 Final Accuracy Testing
The final testing phase required full hardware and software integration, marking the transition of the project into its fully functional state. At this stage, the machine learning model was compressed into a lightweight header file utilized by the ESP32-C6's flashed code. Once the model was integrated into the system, sensor data was able to be gathered and stored into grouped files for model training. The testing process was conducted by gathering 5 files per defined gesture, followed by retraining the model with the updated sensor information. Each time the model was trained, 20% of the files were separated and used to verify the model's accuracy via testing. The model’s accuracy was noted at each stage, and if it remained below an acceptable threshold, the cycle repeated, where 5 more datasets per gesture were recorded along with the model being trained again. 

Figure 18 – Accuracy results at each training stage
Figure 18 displays the progression of model accuracy as additional data was incorporated. During testing, we observed that the LSTM model’s file size increased with the growing number of gestures, eventually exceeding the memory constraints of the ESP32-C6. Because the microcontroller could not store the expanded LSTM model, we transitioned to a 1-Dimensional Convolutional Neural Network. While the 1D-CNN maintained a manageable file size suitable for the device, this change introduced a slight reduction in accuracy—a trade-off we determined to be acceptable for deployment on the hardware.
For comparison purposes, both models were trained to the full dataset size of 390 gesture samples (30 training files per gesture). At this final stage, the LSTM model achieved 98% accuracy, while the 1D-CNN achieved 95%. Both results exceeded our performance target of 75% gesture-recognition accuracy. Therefore, the 1D-CNN model was selected for deployment, and the gesture-detection subsystem was considered ready for full integration with the rest of the device.
6.2.4 Latency Testing
The primary objective of the latency test was to ensure that the HandSense system’s “gesture-to-action” latency was low enough that the user would have a smooth interaction while using the glove. The testing was done after final software and hardware integration. The average latency that we recorded through the serial monitor from sensor data reading to sending the model inference gesture to the glove receiving the acknowledgement was 111 milliseconds as shown in figure 19. This value was collected from 20 samples. It was found that the biggest delay in latency was the Bluetooth communication between the gloves and the PC. The latency could be improved if the Bluetooth protocol was updated to use the faster Bluedroid Bluetooth stack instead of using the lighter weight NimBLE stack.
